# Контроль команды

Несколько типовых сценариев директора.

## Посмотреть, чем занимается конкретный менеджер

`/dashboard/requests` → фильтр «**по менеджеру**» (выпадающий список вверху таблицы, виден только привилегированным ролям). Выбираете коллегу → видите все его открытые заявки в одной выдаче.

Открыть карточку → видны все табы: Переписка, Позиции, КП, Счета, Активность. **Менять статусы нельзя**, потому что директор не assignee. Но видно всю историю.

## Разобраться, почему заявка попала не к тому

В карточке заявки → таб **«Активность»** → запись `request_assigned` содержит `reason`:

- `auto_sticky:{"kind":"catalog","linked":[1234,1235]}` — sticky-правило по `catalog_item_id`. В `linked` — другие заявки менеджера с этим же товаром.
- `auto_sticky:{"kind":"client","linked":[…]}` — sticky по email клиента.
- `auto_round_robin:{"loads":{16:34,17:33,…},"min_load":33,"lru_pool":[17]}` — взяли по min(effective_load) + LRU tiebreak.
- `manual_reassign: <текст РОПа>` — переподчинение вручную.

Если решение системы выглядит странно — обычно проблема в одном из:
- Sticky правило сработало по «случайному» совпадению.
- Reply пришёл на letter не от того клиента (linker ошибся).
- Менеджер был в отпуске, ушло дежурному.

## Проверить, что AI не теряет заявки

`/dashboard/mail-review` — все письма, отклонённые AI как irrelevant. Просмотрите фильтром «7d», ищите по теме / email клиента.

Если видите реальный клиентский запрос — кнопка «Это заявка». Создастся Request, дальше идёт обычный pipeline.

Если AI стабильно ошибается на каком-то паттерне (тема / отправитель / шаблон) — попросите РОПа создать [правило почты](/docs/rop/mail-rules) с `action=trigger_request_creation` для этого паттерна.

## Контроль нагрузки

Дашборд → «Загрузка менеджеров». Если один менеджер постоянно > 100%, а другие < 60% — два варианта:

1. **Переподчинить** часть заявок (через РОПа): открыть карточку перегруженного → «Переподчинить» вручную.
2. **Снизить плановую нагрузку** перегруженного через `/dashboard/managers/{id}/edit` → «Плановая нагрузка %» → 80 (или ниже). Эффект постепенный — система будет реже выбирать его для новых заявок, пока average не выровняется. См. [Распределение](/docs/rop/distribution).

## Контроль OpenAI

В шапке появилось красное «⛔ OpenAI недоступен» в bell-колокольчике — значит circuit-breaker открылся (3+ подряд 429/timeout у категоризатора). Кликаете → попадаете на платёжку OpenAI.

После пополнения первый успешный запрос автоматически снимает паузу, scheduled `mail:categorize --all` каждые 5 минут добивает накопленный backlog. Никаких действий в системе не нужно — только проверить через 10-15 минут, что новые письма снова получают категорию (в разделе «Почта» столбец «КАТЕГОРИЯ»).
