# Где что — навигация Директора

Директорат имеет почти полный набор разделов как у РОПа, но **операционно не работает с пулом заявок** — фокус на наблюдение, аналитику и стратегические решения.

## Шапка

| Элемент | Что делает |
|---|---|
| **Workspace-пилл** `mail@…` | Состояние общих ящиков системы. |
| **Дашборд** | Полная версия (см. [Дашборд](/docs/director/dashboard)). |
| **Заявки** | Все заявки всех менеджеров. По умолчанию scope=`all` (не «мои» — у директора заявок не бывает). |
| **Правила почты** | Просмотр и редактирование. |
| **Авто-отклонённые** | Список + возможность реоткрыть как заявку (но обычно это работа РОПа/секретаря). |
| **Менеджеры** | Список + редактирование. |
| **Настройки** | Override параметров. |
| **❓ / ▲ / аватар** | Как у всех. |

## Чем директор отличается от РОПа

Технически — почти ничем (`role:head_of_sales,director,admin` для одних и тех же роутов). По смыслу:

- РОП **управляет операционно** — переподчиняет, разбирает спорные кейсы, настраивает правила.
- Директор **смотрит и принимает решения** — куда вкладываться, кого нанимать, какие KPI правильные.

В пуле директор по умолчанию видит **все** заявки (scope=all автоматически на mount'е, если в URL не указано иное), потому что фильтр «свои» для него возвращает 0.

## Что директор НЕ делает

- Не назначается на конкретные заявки (`Role::requestHandlerRoles` не включает `director`).
- Не подключает личный ящик к системе (нет смысла — не получает заявок).
- Не управляет общими ящиками `info@/mail@` (только админ).

## Что обычно нужно директору в день

1. **Дашборд** — общая картина за период (7/30/90 дней).
2. **Заявки → фильтр по менеджеру** — посмотреть как у конкретного коллеги.
3. **Авто-отклонённые** — проверить что AI не теряет важное.
4. **Менеджеры** — посмотреть отпуска и нагрузку команды.

См. также [Контроль команды](/docs/director/oversight) — какие сценарии и как ими пользоваться.
