# Правила почты

`/dashboard/mail-rules` — кастомные правила, которые применяются к входящим письмам **после** AI-категоризатора. Это «жёсткая» автоматика для случаев, где нельзя полагаться на LLM.

## Когда нужны кастомные правила

- Письма с конкретного домена (например, `liftway.store`) — всегда `client_request`, без LLM.
- Тестовые письма от вашего тестового адреса — `irrelevant`.
- Письма от поставщика — пометить лейблом, но Request не создавать.
- Перенаправить определённый тип писем на дополнительный inbox (forward).

## Структура правила

| Поле | Что задаёт |
|---|---|
| **Название** | Свободная строка для опознания в списке. |
| **Активно** | Включить/выключить без удаления. |
| **Терминальное** | Если match — дальнейшие правила и AI-категоризатор НЕ запускаются. По умолчанию `true`. |
| **Match mode** | `any_of` (любой критерий совпал) или `all_of` (все). |
| **Критерии** | Список условий: поле + оператор + значения. |
| **Действие** | Что сделать при совпадении. |

### Критерии

Каждый критерий — это тройка:

| Поле (`field`) | Оператор (`operator`) | Значения |
|---|---|---|
| `subject` — тема письма | `contains_any` — содержит любое | список строк |
| `from_email` — полный адрес | `not_contains` — НЕ содержит | список строк |
| `from_domain` — только домен | `equals_any` — точное совпадение | список строк |
| `body` — тело письма | `regex_match` — регулярное выражение | regex-паттерны |
| | `ends_with` — заканчивается на | список окончаний |

### Действия

| Action type | Что делает |
|---|---|
| `label_only` | Просто навешивает label на письмо. Дальше pipeline идёт обычным путём (категоризатор / linker / processIfRequest). |
| `trigger_request_creation` | Принудительно создать `Request` из этого письма независимо от того, что скажет AI-категоризатор. Используйте для известных «трудных» клиентов, у которых формат письма ломает LLM. |
| `forward` | Переслать письмо на указанный адрес. Дальше pipeline продолжается (если не `isTerminal`). |

## Создание правила

`/dashboard/mail-rules/create` — пошагово:

1. Имя, активность, флаг «терминальное».
2. Match mode + критерии (минимум один).
3. Действие.
4. Сохранить.

## Порядок применения

`MailRoutingRuleEngine::match()` идёт по правилам в порядке `id ASC`. Первое сработавшее правило с `isTerminal=true` останавливает дальнейшую обработку. Не-терминальные применяются все по очереди.

## Что **не** делает движок правил

- Не классифицирует **что это за письмо** (request / reply / spam) — это работа AI-категоризатора.
- Не выбирает менеджера для созданной Request — это AssignmentService (sticky + round-robin).
- Не парсит позиции — это ParseRequestItemsJob после создания Request.

Правила — это «надстройка» для override AI там, где он систематически ошибается.
