# Настройки

`/dashboard/settings` — override параметров системы. Все значения имеют fallback на `config/services.php` + `.env`, поэтому пустое значение = «использовать default». Изменения видны на следующий запрос (без `queue:restart`).

Все настройки сгруппированы.

## Каталог · матчинг по name

Семантический матчинг позиций заявки с каталогом через эмбеддинги (когда нет точного совпадения по `M-SKU` или `brand_article`).

| Ключ | Что задаёт |
|---|---|
| **Включить C-step** | Глобальный killswitch семантического матчинга. Выключить — позиции без точного артикула не сматчатся, придётся вручную. |
| **Порог cosine similarity** | Минимум сходства (0..1) чтобы кандидат вообще рассматривался. По умолчанию 0.75. Меньше → больше ложно-положительных. |
| **High-confidence порог** | Если similarity ≥ этого значения — LLM-валидация пропускается, матч принимается «как есть». По умолчанию 0.90. |
| **LLM-валидация** | Включить/выключить gpt-4o-mini проверку «один ли это товар». Полезно отключить при выработанной OpenAI-квоте. |
| **Поведение при сбое LLM** | `reject` (default, отбросить кандидата — приоритет precision) или `accept` (принять без проверки — приоритет recall). |

## Каталог · импорт

| Ключ | Что задаёт |
|---|---|
| **Минимум строк в full snapshot** | Защита от обнуления каталога: если 1С прислал full snapshot меньше N строк — отвергаем. После первой нормальной выгрузки имеет смысл повысить до ~80% от ожидаемого размера, чтобы битая частичная выгрузка не схлопнула таблицу. |

## Налоги

| Ключ | Что задаёт |
|---|---|
| **Ставка НДС, %** | Используется для расчёта итога по каталожным ценам в карточке заявки. По умолчанию 22 (с 2026). До 2026 — 20. Поддерживает дробные значения. |

## Распределение заявок

| Ключ | Что задаёт |
|---|---|
| **Скорость догона отстающих (X)** | Параметр weighted-random алгоритма (legacy, сейчас в коде используется детерминированный least-loaded). Рекомендуемый диапазон 1.5..3.0. |
| **Порог автопометки дилера** | Сколько открытых заявок у одного `client_email` нужно, чтобы автоматически пометить его как дилерский (пропускать sticky уровня «по email клиента»). По умолчанию — большое число, рассчитанное на реальный поток. |

## Attention · дедлайны

Сколько рабочих часов/дней может пройти без действия по заявке в каждом статусе, прежде чем поднимется `sla_breach`:

| Ключ | Default | Что задаёт |
|---|---|---|
| Новая → действие | 4 ч | Менеджер должен открыть карточку. |
| Назначена → менеджер открыл | 8 ч | Открыли, не приступили. |
| В работе без активности | 24 ч | Никаких изменений за день — нудж. |
| Жду клиента — первый ремайндер | 3 дн | Клиент молчит — пора напомнить. |
| КП отправлено — первый нудж | 3 дн | Клиент думает — пора уточнить. |
| На согласовании | 5 дн | «Обсуждаем внутри» — нужно надавить. |
| Ждём счёт | 8 ч | Клиент согласился, счёт не выставили. |
| Счёт отправлен — нудж | 3 дн | Ждём оплату. |

Все значения — в **рабочих** часах/днях (Пн-Пт 9-18 по Москве с учётом российского производственного календаря).

## DocumentDetector · auto-mode

Распознавание исходящих КП/счетов в Sent-папке.

| Ключ | Что задаёт |
|---|---|
| **Минимальная confidence для auto-apply** | По умолчанию 0.85. Если детектор уверен ≥ этого порога — переход статуса заявки происходит автоматически. Ниже — оператор видит «AI: Отправлено КП, уверенность 60%», переход вручную. |
| **Auto: КП → quoted** | Включить/выключить auto-apply для `outbound_quotation_full`. |
| **Auto: Счёт → invoiced** | Включить/выключить auto-apply для `outbound_invoice`. Дополнительно создаётся `Invoice` (для таба «Счета» и общего раздела). |
