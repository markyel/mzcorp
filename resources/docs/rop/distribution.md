# Распределение заявок

Когда приходит новое письмо клиента и из него создаётся `Request`, система выбирает менеджера по двум уровням: **sticky** (привязка к истории) → **round-robin** (балансировка нагрузки).

## Sticky-правила

Проверяются по убыванию надёжности сигнала, первый сработавший побеждает:

| Уровень | Правило | Когда побеждает |
|---|---|---|
| **0. Личный ящик** | Письмо пришло на personal-mailbox менеджера. | Клиент написал лично — owner становится assignee, даже если он сейчас в отпуске (delegation покроет). |
| **1. По каталогу** | У позиций новой заявки уже резолвлен `catalog_item_id`, такой же есть в открытой заявке у менеджера. | «Один товар → один менеджер». |
| **2. По email клиента** | У менеджера есть открытая заявка от того же `client_email`. | «Один клиент → один менеджер». |
| **3. По артикулу/названию** | Текстовый матч на `parsed_article` / `parsed_name` (когда каталог ещё не сматчился). | Fallback на сырые поля. |

Sticky всегда побеждает балансировку.

### Исключение: дилерские email'ы

Если у `client_email` уже **≥ N открытых заявок** в системе (порог `dealer.auto_threshold` в [Настройках](/docs/rop/settings)), email автоматически помечается как «дилерский» и **пропускает уровень 2 «по email клиента»**. Поток дилера распределяется через round-robin, чтобы не топить одного менеджера.

Уровни «по каталогу» и «по артикулу» для дилеров продолжают работать.

## Round-robin

Когда sticky ничего не нашёл, заявка идёт **тому, у кого минимальная effective load**.

`effective_load = current_open_count / (load_weight / 100)`

- Менеджер с `load_weight=100` (норма), 10 открытых → effective=10.
- Менеджер с `load_weight=200` (план ×2), 10 открытых → effective=5 → получит следующую.
- Менеджер с `load_weight=50` (план ½), 4 открытых → effective=8.

Tiebreak при равных effective — **LRU**: кому давнее назначали, тот первым в очереди. NULL `last_assigned_at` (никогда не получал) — самым первым.

Плановую нагрузку каждого менеджера РОП меняет в его карточке — см. [Менеджеры](/docs/rop/managers).

## Переподчинение

В карточке заявки → **«Переподчинить»**. Выбираете нового assignee из списка request-handlers. Старый менеджер теряет заявку из пула, новый получает в attention.

Audit-запись пишется в `request_assignments` с `reason=manual_reassign + комментарий`.

### Делегации (отпуск коллеги)

Когда менеджер уходит в отпуск (поле `unavailable_from / unavailable_until` в его профиле), его активные заявки **автоматически передаются дежурному** через `RequestDelegation`. Дежурный видит чужие заявки в своём пуле с пометкой «временно от @Имя». В день возвращения заявки откатываются обратно.

Конфигурация: в карточке менеджера флаг `unavailable_auto_delegate`.

## Attention

`AttentionService` поднимает заявки наверх пула с пометкой когда от менеджера ждут действия:

| Reason | Когда |
|---|---|
| `fresh_assignment` | Только что назначена, ещё не открывали. |
| `client_replied` | Клиент ответил по треду. Amber-подсветка, **не алярм** (не в bucket «Просрочено»). |
| `sla_breach` | Превышен дедлайн SLA по текущему статусу. Это уже алярм. |
| `paused_resumed` | `paused_until` истёк, заявка вернулась в работу. |
| `parser_pending` | Парсер ещё не отработал (для нового кейса). |

Attention снимается, как только менеджер открыл заявку, сменил статус или отправил ответ.

Дедлайны по каждому статусу настраиваются в [Настройках → Attention · дедлайны](/docs/rop/settings).
