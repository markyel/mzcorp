# MyLift — автоматизация работы отдела продаж лифтовых запчастей

## Что это и зачем

**CRM-прослойка** между входящими email-заявками клиентов и **корпоративной базой**, которая является системой записи для КП, ассортимента/цен и счетов. MyLift не подменяет корп. базу и не управляет бизнес-документами — он наводит порядок в процессе обработки заявок до и вокруг создания этих документов.

**Что делает MyLift:**
- Собирает входящие заявки из общих и личных корпоративных ящиков.
- Распределяет заявки между менеджерами по правилам справедливости и стикинеса.
- Помогает менеджеру: показывает каталог (read-only из корп. базы), запрашивает актуализацию цен у поставщиков.
- Мониторит переписку (входящую и исходящую), детектирует ключевые события (КП отправлено, счёт отправлен).
- Контролирует SLA, шлёт напоминания, ведёт follow-up с клиентами и поставщиками.
- Экспортирует summary заявки в корп. базу для архивации.

**Что MyLift НЕ делает:**
- Не генерирует КП (создаются менеджером в корп. базе или внешнем инструменте).
- Не выставляет счета (создаются в корп. базе).
- Не управляет каталогом/ценами/ассортиментом (всё в корп. базе, MyLift только читает).
- Не является «системой правды» по бизнес-документам.

**KPI, которые двигаем:**
- Время от заявки до отправленного КП (target — часы, не дни).
- Доля безответных заявок → стремится к 0.
- Конверсия КП → счёт → оплата (через системные напоминания, а не через память менеджера).

**Главное отличие от классической CRM** — менеджер не «ведёт клиента вообще», а получает поток заявок, распределённый алгоритмом; при этом исторические связки «менеджер ↔ оборудование/клиент» сохраняются.

## Архитектурная позиция

```
[Email клиента]
     ↓
[Yandex 360 ящики (общие + личные менеджеров)]
     ↓ IMAP
┌─────────────────────────────────────┐
│            MyLift (CRM)             │
│  • Приёмка/распределение/SLA       │
│  • Refresh цен у поставщиков        │
│  • Tracking переписки (in+out)      │
│  • Детектор КП/счетов в outbound    │
│  • Напоминания и follow-up          │
└─────────────────────────────────────┘
     ↓ ↑                       ↓ ↑
     ↓ ↑ (read catalog,        ↓ ↑ (export request summary,
     ↓ ↑  optional import КП)  ↓ ↑  optional import КП/invoice metadata)
┌─────────────────────────────────────┐
│   Корпоративная база (1С/иное)      │
│  • Каталог, ассортимент, цены        │
│  • КП (выданные)                    │
│  • Счета                            │
└─────────────────────────────────────┘
     ↑
     ↑ (Менеджер создаёт КП/счёт здесь напрямую,
        либо через интегрированный инструмент)
```

**Поток данных:**
1. **Корп. база → MyLift (read)**: каталог, ассортимент, цены, статусы. Постоянная синхронизация.
2. **MyLift → корп. база (write)**: summary заявки на финальном этапе (закрытие — won/lost). Возможно — несколько checkpoint'ов.
3. **Email → MyLift (capture)**: входящие заявки + исходящие от менеджеров (в т.ч. с КП/счетами во вложениях).
4. **MyLift → email (send)**: refresh-запросы поставщикам, напоминания клиентам.

## Чем отличается от LazyLift

| Аспект | LazyLift | MyLift |
|---|---|---|
| Источник предложения | Внешние поставщики, рассылка запросов | Свой склад + свой каталог (читается из корп. базы) |
| Главный outbound-flow | Письма поставщикам → парсинг ответов → КП | Лукап в каталоге → менеджер делает КП в корп. базе → отсылает клиенту → MyLift детектирует факт отправки |
| Система записи (system of record) | Внутри LazyLift | **Корпоративная база** (1С/иное) — MyLift это CRM-прослойка |
| Создание КП и счетов | В системе | В корп. базе менеджером, MyLift только трекает и экспортирует summary |
| Email-инфраструктура | Gmail через n8n | **Yandex 360 напрямую из Laravel** (IMAP/SMTP), без n8n |
| Tracking исходящих | Не было | **Есть** — мониторим Sent-папки всех ящиков, видим письма, отправленные напрямую из почтового клиента |
| Пользователи | Один оператор / куратор KB | 3 роли: менеджер / РОП / директорат |
| Логика назначения | Не нужна (один оператор) | **Центральная фича**: справедливое распределение + sticky |
| Доменная сущность «Поставщик» | Есть, ключевая (запрос по каждой заявке) | **Есть, но второстепенная** — запрашивается изредка для актуализации цен в каталоге, не по каждой заявке |
| Dispatch к поставщикам | Per-request consolidated dispatch по релевантным поставщикам | **Полностью аналогичный механизм** — рассылка по релевантным поставщикам, парсинг ответов; но триггер другой — актуализация каталожной цены, а не сбор предложений по конкретной заявке клиента |
| KB-модуль | Оценка достаточности заявки для рассылки | Оценка достаточности **+ матчинг на каталог** |
| Напоминания | Базовый модуль есть | Расширяем под новые триггеры (refresh, разблокировка заявки, нудж клиенту по КП, истечение счёта) |
| Метрики | Покрытие KB, качество заявок | Производительность менеджеров (для ЗП), SLA, конверсия |

## Роли и права доступа

| Роль | Видит | Может |
|---|---|---|
| **Менеджер** (5-6 в команде) | Свой пул заявок, свои КП, свои счета, свой каталог (read), статусы актуальности цен, общий пул запросов к поставщикам | Работать с заявкой (отвечать клиенту, формировать КП в 1С, выставлять счёт), запрашивать уточнения у клиента, **запрашивать актуализацию цен у любых поставщиков (по всем заявкам)**, помечать «не профильно» |
| **РОП** (1 в команде) | Все заявки, кому назначены, статусы, дашборд по менеджерам, активные price-refresh batch'и, каталог целиком | Переназначать, разруливать конфликты, **ставить менеджера в «недоступен» с переподчинением открытых заявок** (см. 3.5), корректировать правила распределения, редактировать каталог поставщиков, настройки системы (дедлайны, пороги, шаблоны) |
| **Секретарь** (новая роль) | Карта распределения заявок (кто какие получил), сводка по auto-routing | Контролирует правильность распределения, эскалирует РОПу аномалии. Не настраивает систему, не работает с заявками напрямую |
| **Директорат** | Все заявки + сводные метрики (выручка, конверсии, средний чек, время ответа) + **полный KB-интерфейс** | Просмотр + экспорт. **Также выполняет роль куратора KB**: редактирует категории, бренды, параметры, экстракторы, разметка по примерам. |
| **Поставщик** (внешний, не пользователь системы по умолчанию) | — | Получает email с запросом актуализации цен, отвечает письмом. На старте — email-only без личного кабинета |

> **Каталогист/Закупка** на старте отдельной ролью не выделяется — функции по управлению каталогом и поставщиками лежат на РОПе.
>
> **KB-куратор** — отдельной ролью не выделяется, объединена с директоратом.
>
> **Расчёт ЗП менеджеров** в систему **не входит** — это вне функционала проекта. Метрики производительности доступны для дашбордов, но используются для operational контроля, а не для расчёта вознаграждений.

Реализация — `spatie/laravel-permission`, Laravel Gates/Policies, проверки в Livewire-компонентах и middleware.

## Архитектурные слои

### 1. Email integration (новый ключевой модуль `app/Services/Mail/`)

**Полностью заменяет n8n из LazyLift.** Laravel напрямую работает с Yandex 360 через IMAP (приём) и SMTP (отправка). Подключаемся к двум типам ящиков:

- **Общие ящики** (`sales@..`, `info@..`) — основной поток новых заявок.
- **Личные ящики менеджеров** — заявки, которые клиенты пишут конкретному менеджеру, и **исходящие письма менеджеров** (в т.ч. написанные напрямую из почтового клиента, минуя MyLift).

**Стек:**
- Библиотека: `webklex/laravel-imap` (pure PHP, не требует расширения `imap`).
- Отправка: дефолтный Laravel Mail (Symfony Mailer) с per-mailbox SMTP transport.
- Yandex 360 endpoints: `imap.yandex.ru:993` SSL, `smtp.yandex.ru:465` SSL.
- Аутентификация — на старте **App passwords** (менеджер генерирует в настройках Яндекса, отдаёт админу), на будущее — **OAuth 2.0** через Yandex 360 API (требует регистрации приложения).
- Папки Yandex по-русски (`Входящие`, `Отправленные`, `Спам`, `Корзина`) — мониторим через IMAP special-use flags (`\Inbox`, `\Sent`), не по именам.

**Что мониторим в каждом ящике:**

| Папка | Что делаем | Зачем |
|---|---|---|
| `\Inbox` | Парсим, классифицируем (AI), создаём/обновляем `Request` | Основной приём заявок и переписки от клиентов |
| `\Sent` | Парсим, привязываем к существующей `Request` по `Message-ID`/`References` | **Tracking исходящих** — менеджер пишет напрямую из Яндекс веб-клиента, мы это видим |
| `\Spam`, `\Trash` | Не трогаем | — |

**Обработка входящего письма** (pipeline `IncomingMailProcessor` → `MailDispatcher`):

```
1. fetchNewMessages(mailbox, folder) — UID-инкрементальный fetch
2. для каждого Message:
   2.1 dedup: SELECT FROM email_messages WHERE message_id = ?
   2.2 сохранить EmailMessage (raw headers, body, body_html, attachments)

   2.3 ПРИМЕНЕНИЕ ПРАВИЛ МАРШРУТИЗАЦИИ (см. 1.5):
       - Прогоняем настроенные mail_routing_rules в порядке priority
       - Первое match → выполняем action (forward + label) → exit
       - Если ни одно не сработало → AI-классификация типа письма

   2.4 AI-классификация типа (gpt-4o-mini):
       request | reclamation | accounting | general_question | spam | other

   2.5 Действие по типу:
       - request → создать Request + RequestItem(ы), L1/L2 KB → AssignRequestJob
       - reclamation/accounting/general → дефолтное правило для типа (forward + label)
         или fallback (просто метка "не разобрано", алерт секретарю/РОПу)
       - spam → метка "spam", не трогаем
       - reply (если нашли In-Reply-To на существующий Request) → привязать как thread item

   2.6 ВСЕГДА: ставим IMAP custom label на исходное письмо
       (НЕ ставим \Seen — для пользователя письмо остаётся непрочитанным)

3. сохранить state: last_uid_seen, uid_validity per mailbox per folder
```

#### 1.5 Mail routing rules — правила маршрутизации входящих

Не вся почта в общих ящиках — заявки. Часто прилетают рекламации, бухгалтерские вопросы, общие вопросы, маркетинговые письма. MyLift даёт **настраиваемые правила** перенаправления, чтобы письма попадали к тем, кому надо, без участия секретаря.

**Модель `MailRoutingRule`:**

```
- id, name (например, "Рекламации")
- priority (int, порядок применения, ниже = раньше)
- is_active (bool)
- mailbox_scope (jsonb) — на какие ящики применять (или все)
- match_mode enum: any_of | all_of | ai_classified
- match_criteria (jsonb), массив условий:
    - {field: "subject", op: "contains_any", values: ["рекламация", "претензия"]}
    - {field: "from_domain", op: "equals", values: ["company.ru"]}
    - {field: "body", op: "contains_any", values: ["возврат", "брак"]}
- ai_match_type (опц., если match_mode = ai_classified): "reclamation" / "accounting" / etc.
- action_type enum: forward | label_only | trigger_request_creation
- forward_to_email (опц.)
- forward_template_id (опц., FK на EmailTemplate — для composer-style forward с пояснением)
- label (string, имя IMAP custom-label, например "MyLift/Рекламации")
- is_terminal (bool) — если true, после применения этого правила больше не проверяем
- created_by_user_id, updated_at
```

**Примеры правил (РОП настраивает в UI):**

| Имя | Критерии | Action | Label |
|---|---|---|---|
| Рекламации | subject содержит "рекламация" / "претензия" / "брак" / "возврат" | forward → claims@company.ru | `MyLift/Рекламации` |
| Бухгалтерия | subject/body содержит "акт сверки" / "счёт-фактура" / "оплата" / "баланс" | forward → buh@company.ru | `MyLift/Бухгалтерия` |
| Заявка | AI classified as `request` | trigger Request creation | `MyLift/Заявка` |
| Маркетинговый шум | from_domain ∈ blacklist | label_only (тихо отметить) | `MyLift/Маркетинг` |
| Не разобрано (fallback) | catch-all | label_only + alert РОПу | `MyLift/Не разобрано` |

**UI настройки** `/dashboard/mail-rules` (роль РОП):
- Список правил с drag-n-drop приоритета.
- Редактирование критериев, выбор label из существующих или создание новой.
- **Превью**: РОП может выбрать любое реальное письмо из последних N → MyLift показывает «к этому письму применилось бы правило X, действие Y». Для тестирования.
- Метрики per rule: сколько писем за период подпали под это правило.

#### 1.6 IMAP-метки (labels) для секретаря

> **Прочитано НЕ ставим.** Секретарь продолжает видеть в Yandex веб-клиенте все непрочитанные письма с их обычным состоянием. Но **дополнительно** на каждое письмо вешаем custom IMAP-флаг (label).

**Зачем:** секретарь, открывая обычный почтовый клиент Яндекс, видит у каждого письма метку и сразу понимает: «это разобрано MyLift'ом и ушло в бухгалтерию», «это — заявка, в работе у Иванова» и т.д. Не нужно дублировать работу.

**Как технически:**
- Webklex/laravel-imap поддерживает `Message::addFlag()` для произвольных custom keywords.
- Yandex 360 IMAP принимает custom flags (формат `$Label-name` или просто `MyLift/...` — зависит от клиентского представления, проверить на боевом ящике).
- Сначала FETCH в READ-ONLY (наша обычная синхронизация), затем отдельной SELECT в read-write — `STORE +FLAGS (\$Label-name)`.
- Не ставим `\Seen`, явно — иначе письмо станет «прочитанным» в Яндексе.

**Стандартный набор labels** (создаётся миграцией, дальше расширяется РОПом через правила):
- `MyLift/Заявка/Иванов` — превратилось в Request, передано Иванову
- `MyLift/Заявка/Петров` — передано Петрову
- `MyLift/Уточнение/Иванов` — привязано к существующей заявке Иванова
- `MyLift/Рекламации` — переслано (метка без менеджера)
- `MyLift/Бухгалтерия` — переслано
- `MyLift/Спам` — спам
- `MyLift/Не разобрано` — fallback (требует ручной обработки)

> **Важно:** для заявок и связанной переписки метка содержит **имя менеджера-исполнителя**. Это даёт секретарю мгновенную видимость в Яндекс веб-клиенте: «это в работе у Иванова», «это передано Петрову». При смене менеджера (переподчинении) — метка обновляется на исходных письмах.

**Динамическое обновление меток:**
- При назначении заявки → метка с именем менеджера ставится на исходное входящее письмо.
- При переподчинении → старая метка удаляется, новая ставится.
- Опционально (фаза 2): расширенная метка со статусом — `MyLift/Заявка/Иванов/КП-отправлено`.

> Для писем, не превращающихся в заявку (форварды на бухгалтерию и т.п.), метка статичная и не содержит менеджера.

**Обработка Sent-папки** (pipeline `OutgoingMailObserver`):

```
1. fetchNewMessages(mailbox, '\Sent')
2. для каждого Message:
   2.1 dedup по message_id (письма, отправленные через MyLift, тоже сюда попадут — не плодим)
   2.2 пытаемся привязать к Request:
       - по In-Reply-To → найти inbound с этим Message-ID
       - по References → проверить thread
       - по получателю → последняя открытая заявка от этого клиента у этого менеджера
   2.3 если нашли Request → создать EmailMessage (direction=outbound, source=external_client)
       и обновить request.last_outbound_at (для метрики ответа)
   2.4 если не привязали → положить в "висящие" outbound для ручного матчинга РОПом
```

**Real-time vs polling:**
- Старт — polling каждые 1-2 минуты через scheduled job `mail:sync` (по одному job на ящик, параллелизуется через очереди).
- При росте — IMAP IDLE (`webklex/laravel-imap` поддерживает) через долгоживущий Supervisor-процесс на mailbox. Решаем на основе боли.

**Идемпотентность и устойчивость:**
- Уникальный ключ — `Message-ID` (заголовок) + `mailbox_id` + `folder` (одно письмо может лежать в Inbox у получателя и в Sent у отправителя — это разные `EmailMessage`, но `request` связь общая).
- IMAP fetch для чтения — в **READ-ONLY** режиме. Для постановки label — отдельная SELECT в read-write режиме, операция `STORE +FLAGS`. **`\Seen` явно не ставим**.
- Учёт `UIDVALIDITY`: если изменилось — full resync с этой папки (Yandex это делает редко, но при миграциях бывает).
- Сбой подключения — exponential backoff, не валим job, в Sentry.

### 3. Распределение заявок (новый модуль `app/Services/Assignment/`)

**Сердце системы.** Алгоритм запускается job'ом сразу после создания заявки и L1/L2 обработки KB.

```
AssignRequestJob:
  1. Извлечь сматченные позиции каталога (через KB → CatalogMatchService)
     -> matched_catalog_item_ids = [...]

  2. Sticky-проверки в порядке убывания приоритета:
     a) personal_mailbox: заявка пришла на личный ящик менеджера
        -> назначить ему (если активен)
        (это и есть «клиент-привязка» — клиенты привязанных
         менеджеров пишут им напрямую на личный адрес)

     b) catalog_item_sticky: есть открытая заявка с одной из
        sматченных catalog_item_ids
        -> назначить владельцу (без cap — см. ниже)

  3. Если ни одно sticky не сработало — load balancing:
     candidates = active_managers()
     weights = current_workload(candidates)
     chosen = min_weight(candidates)
     -> назначить chosen

  4. Записать reason в request_assignments (audit log).
```

**Параметры алгоритма** (хранятся в `Setting`, редактируются РОПом):
- `period_for_fairness`: `week | month` — за какой период считать «equal volume» для тай-брейка

**Что упрощено:**
- **Cap нагрузки на менеджера НЕ задаётся** — sticky всегда побеждает. РОП видит распределение в дашборде и при необходимости вручную перераспределяет.
- **Клиент-sticky отдельно не моделируется** — реализуется через факт, что привязанные клиенты пишут менеджеру на его личный ящик (правило 2a).
- **«Что считать одним оборудованием»** — точное совпадение `catalog_item_id` после матчинга. Если позиция не сматчилась с каталогом — sticky по этой позиции не работает, идём в load balancing.

**Метрика workload** для распределения — кол-во активных заявок (статусы `in_progress`, `awaiting_clarification`, `quoted_no_answer`, `pending_price_refresh`).

> Метрики «производительность за период» используются для operational дашборда, **не для расчёта ЗП** (расчёт ЗП — вне функционала проекта, см. раздел «Роли»).

#### 3.5 Менеджер «недоступен» с переподчинением заявок

Когда РОП ставит менеджера в статус «недоступен» (отпуск, болезнь), интерфейс требует **сразу решить, что делать с открытыми заявками этого менеджера**:

```
[РОП нажал "сделать недоступным"] 
  → modal: 
     "У менеджера X есть N открытых заявок. Что делать?"
     ○ Оставить за ним (вернутся когда выйдет)
     ○ Переподчинить вручную одному менеджеру: [select]
     ○ Распределить автоматически по правилам (без catalog_item_sticky к нему)
     [Подтвердить]
```

При автоматическом перераспределении:
- Алгоритм запускается заново для каждой заявки.
- Catalog_item_sticky к недоступному менеджеру игнорируется.
- Записи в `request_assignments` фиксируют переход с reason = `manager_unavailable`.

### 4. Каталог + актуализация цен (новый модуль `app/Domain/Catalog/`)

> **Граница ответственности.** Каталог — read-only для MyLift. Источник истины — **корпоративная база**, MyLift синхронизирует данные через `CatalogSyncService` (см. ниже).
>
> MyLift НЕ управляет жизненным циклом цен:
> - Не вычисляет TTL и не помечает цену как `stale`.
> - Не обновляет `current_price` из ответов поставщиков.
> - Не ведёт собственную историю цен.
>
> **Задача MyLift по refresh:** запросить релевантных поставщиков, получить ответы, распарсить, **показать менеджеру**. Дальше менеджер передаёт информацию в корп. базу (вне MyLift).

#### 4.0 Синхронизация с корпоративной базой

`CatalogSyncService` — модуль чтения каталога из корп. базы. Реализация зависит от того, как корп. база предоставляет данные (см. открытые вопросы):

| Способ | Когда подходит | Реализация |
|---|---|---|
| **REST API корп. базы** | Если у корп. базы есть API (1С, кастомное решение) | Scheduled job `catalog:sync`, инкрементально по timestamp |
| **Прямое чтение БД (read replica)** | Если у нас есть доступ к БД корп. системы | Eloquent на отдельном connection, кэш в нашей таблице |
| **CSV/XLSX файл** | Если нет API/доступа к БД | Cron импорт из shared folder |
| **Push от корп. базы** | Если корп. база сама может пушить webhook'и | Webhook endpoint в нашем API |

На старте — выбираем один способ под конкретную корп. базу клиента. Архитектурно изолируем за интерфейсом `CatalogProviderInterface`, чтобы можно было поменять.

`CatalogSyncLog` — таблица синков: когда, что обновилось, ошибки.


#### 4.1 Состояния позиции каталога (что мы читаем)

| Атрибут | Значения | Кто заполняет |
|---|---|---|
| `availability` | `in_stock` / `under_order` / `discontinued` | Каталог (вне MyLift workflow) |
| `price_status` | `actual` / `stale` | Каталог (вне MyLift workflow) |
| `current_price` | decimal | Каталог |

**Подсказка менеджеру при сборке КП:**
> Если позиция со `stale`-ценой — MyLift показывает менеджеру **предупреждение** и предлагает запустить refresh. Это не жёсткий блок: менеджер всё равно может создать КП в корп. базе руками (мы это не контролируем). MyLift просто помогает не упустить устаревшие цены и быстро их обновить, если решено отправлять.
>
> Аналогично для позиций `discontinued` или `out_of_stock` — show, don't block.

#### 4.2 Модели

- `CatalogItem` — позиция: SKU, название, бренд (FK на `ManufacturerBrand`), категория (FK на `EquipmentCategory`), параметры (jsonb), unit, current_price, currency, `availability`, `price_status`. Для MyLift workflow — read-only.
- `Supplier` (наследуется из LazyLift, **расширяется**) — name, email, phone, **assortment_description (text)** — ручное текстовое описание ассортимента и марок, **assortment_matrix (jsonb)** — производное от описания: построенная AI-сервисом матрица «категория × бренд» (массив пар или вложенная структура), **matrix_built_at, matrix_built_with_model**. Кнопка «пересобрать матрицу» (если описание поменяли). При рассылке refresh — пересечение матрицы поставщика с (brand × category) товара.
- `SupplierOffer` (наследуется из LazyLift, **расширяется**) — каждый парсенный ответ поставщика. Поля: supplier_id, catalog_item_id, batch_id, **type enum (`quoted` | `refused`)**, price nullable, currency, quoted_at, valid_until_text (как поставщик написал, без интерпретации), refusal_reason nullable (для type=refused, краткая причина из ответа), raw_text. Может быть несколько per item — по одному на каждого ответившего поставщика. **Это для просмотра менеджером**, не для авто-применения.
- `StockBalance` — если в каталоге есть учёт остатков, мы их видим (read-only).
- `CatalogAlternative` — аналоги/альтернативы (опционально, если нужно подсказывать замены при stale-позиции).

> **Что НЕ моделируем:** `CatalogPriceHistory`, `current_price_supplier_id`, `price_actual_until`, `last_price_refresh_at` — всё это либо в каталоге, либо вообще вне scope.

#### 4.3 Матчинг позиций заявки на каталог

`CatalogMatchService` — для позиции из `RequestItem` ищет кандидатов в каталоге. Использует тот же подход, что KB в LazyLift: extractors + pgvector по `name` и параметрам. Возвращает топ-N с confidence.

В UI заявки менеджер видит подобранных кандидатов (с пометкой статуса цены) и одним кликом добавляет в КП.

#### 4.4 Процесс актуализации цен (`app/Services/PriceRefresh/`)

> **Архитектурно — точная копия dispatch из LazyLift.** Отличия только в семантике (не «сбор предложений по заявке клиента», а «получи ответ для менеджера, чтобы он понимал текущую рыночную цену») и в действии после получения ответа: **только сохраняем `SupplierOffer` и нотифицируем менеджера**, ничего в каталоге не трогаем.

**Триггер — заявка-driven** (единственный сценарий на старте):
- Менеджер собирает КП, видит позицию со `stale`-ценой → блок выдачи.
- Жмёт «запросить актуализацию у поставщиков» по нужным позициям.
- **Подбор поставщиков** (как в LazyLift): для каждой позиции находим всех `Supplier`'ов, у которых `assortment_matrix` (построенная AI из описания) покрывает (бренд позиции) **и** (категория позиции). Это релевантные поставщики. Если матрица устарела (изменилось описание) — система предупреждает об этом РОПа.
- Группируем `Supplier × items[]` — каждому релевантному поставщику уйдёт своё письмо со списком тех позиций из выборки, которые он может покрыть. Один поставщик = один `PriceRefreshBatch`.
- Заявка переходит в статус `pending_price_refresh` с указателем на batch'и.

> **Batch-сценарий (плановое обновление каталога вне заявок) — за рамками проекта.** Если каталог сам управляет TTL, инициатива должна идти оттуда.

**Pipeline:**

```
PriceRefreshService::refresh(items[], request_id):
  1. Для каждой позиции — найти релевантных поставщиков
     (ассортимент покрывает brand AND category)
  2. Развернуть в матрицу supplier × items
  3. Для каждого supplier создать PriceRefreshBatch:
     status: draft → sent → responses_in → closed_by_manager | expired
     items: те позиции, которые этот supplier может покрыть
  4. Сформировать письмо (Blade-шаблон) — список позиций с параметрами
  5. Отправить через SMTP (общий ящик закупок), сохранить Message-ID
  6. request.status = pending_price_refresh,
     связь request ↔ batch'и через request_price_refresh_links

Парсер ответа (триггер — входящее письмо в pipeline IncomingMail):
  1. In-Reply-To / References матчится с PriceRefreshBatch.message_id
  2. AI парсит ответ (gpt-4o, QuoteParsing из LazyLift):
     для каждой позиции в batch'е определяет один из исходов:
     - quoted: дана цена + (опц.) валюта + (опц.) срок действия как текст
     - refused: явный отказ ("не работаем", "нет в наличии", "не поставляем")
     - skipped: позиция не упомянута в ответе вообще (silent skip)
  3. Для каждой позиции — создаём SupplierOffer с type ∈ {quoted, refused},
     обновляем PriceRefreshItem.status соответственно.
     Skipped — оставляем в pending (ждём дальнейших ответов или нудж).
  4. Когда все items в batch обработаны → batch.status = responses_in
     (или partial_responses, если есть skipped).
  5. Notify менеджеру: «по заявке #X пришёл ответ от поставщика Y
     (M из N позиций — цены, K — отказы)»
     (in-app + push)

Закрытие refresh (менеджер):
  - Менеджер видит таблицу: позиция | offers (от каждого ответившего поставщика) | отказы
  - Решает: «достаточно информации, продолжаю» → нажимает «закрыть refresh»
  - request.status возвращается в in_progress
  - batch.status = closed_by_manager
  - Если каталог тем временем обновился (price_status стал actual) — менеджер собирает КП.
    Если нет — менеджер либо ждёт каталог, либо договаривается о ручном обновлении вне MyLift.

Напоминание поставщику (ручное, инициатива менеджера или РОПа):
  - В UI пула или заявки — кнопка «напомнить» по поставщику.
  - Берём все PriceRefreshItem.status = pending от этого поставщика
    (по одной заявке или сразу по всем — зависит от точки входа).
  - Формируем reminder-письмо (отдельный шаблон) со списком этих позиций.
  - Шлём через тот же ящик, как продолжение thread'а (In-Reply-To = original message_id).
  - Обновляем item.last_reminded_at, increment reminder_count.
  - Тот же парсер обработает ответ.
```

#### 4.5 Состояние позиции в batch'е (PriceRefreshItem)

| status | Что значит |
|---|---|
| `pending` | Запрос отправлен, ответа нет (полное молчание или skipped в ответе) |
| `quoted` | Поставщик прислал цену → есть `SupplierOffer` с type=quoted |
| `refused` | Поставщик явно отказал → есть `SupplierOffer` с type=refused |
| `cancelled` | Менеджер отменил запрос (например, передумал) |

Дополнительно на item: `last_reminded_at`, `reminder_count`.

#### 4.6 UI: два уровня просмотра

**a) В карточке заявки** (для менеджера, контекст этой заявки) — блок «Refresh цен»:
- Для каждой позиции в работе:
  - Текущая цена в каталоге и её статус.
  - Сводка по поставщикам: «3 цены, 1 отказ, 2 без ответа».
  - Раскрывашка: список offers (цена / валюта / срок / время ответа / ссылка на исходное письмо), список отказов, список без ответа с кнопкой «напомнить» per supplier.
- Кнопки: «закрыть refresh» (для всей заявки), «напомнить всем не ответившим» (массовое).

**b) Общий пул `/dashboard/supplier-pool`** (для РОПа и менеджеров, агрегированный вид всех активных запросов) — главное **новое представление**:

```
┌────────────────────────────────────────────────────────────┐
│  Pool запросов к поставщикам                               │
│  Фильтры: [поставщик ▼] [статус ▼] [возраст ▼] [мой/все]   │
├────────────────────────────────────────────────────────────┤
│ Поставщик А                              [↻ Напомнить всем]│
│   • 12 позиций ожидают (старейшая: 5 дней)                 │
│   • 8 — цены получены, 3 — отказы, 1 reminder отправлен    │
│   ▼ раскрыть                                                │
│      позиция 1 [pending 3 дня] заявка #LZ-1234 [↻ напомн.]│
│      позиция 2 [pending 5 дней] заявка #LZ-1102 [↻ напомн.]│
│      позиция 3 [quoted 1500₽]   заявка #LZ-1234            │
│      позиция 4 [refused]         заявка #LZ-1102            │
│      ...                                                    │
├────────────────────────────────────────────────────────────┤
│ Поставщик Б                              [↻ Напомнить всем]│
```

Возможности страницы:
- Группировка по поставщику (default) или по позиции/заявке.
- Массовое напоминание per supplier — формирует одно консолидированное письмо со всеми pending позициями этого поставщика (даже если они из разных заявок).
- Точечное напоминание per item.
- Фильтр «только мои» (заявки текущего менеджера) vs все (для РОПа).
- Метрики сверху: общее число pending, среднее время ожидания, топ-3 «молчащих» поставщиков.
- Bulk-действия: выделить N позиций → «отменить запрос» (cancelled status), «переотправить с новым шаблоном».

> Что менеджер делает с полученной информацией о ценах — за рамками MyLift. Возможные сценарии: вручную обновить каталог (через свою каталог-систему), договориться с РОПом, использовать как контекст в переписке с клиентом.

### 5. Жизненный цикл заявки (state machine)

#### 5.1 Карта состояний

```
                                                    ┌──────────────────────────┐
                                                    │   ANY STATE может быть   │
                                                    │   PAUSED (по дате)       │
                                                    └──────────────────────────┘

received
  └─> qualifying (KB L1/L2 + классификация тематики)
        ├─> rejected_off_topic ─────────────────────────────────────────► closed_lost
        │     ▲ (уведомление клиенту авто; РОП может reopen → qualifying)
        │
        └─> clarifying_with_client (нужны уточнения)
              ├─ если только часть позиций неясна:
              │    параллельно создаём заявку в режиме "partial" → assigned
              │    менеджер работает с готовыми позициями, дозапрашивает остальные
              │
              ├─> client_replied ─> qualifying (повторная оценка)
              ├─> client_silent (T+N дней без ответа):
              │     auto reminder T+1 ─> auto reminder T+3 ─> closed_lost_no_client_response
              │     ▲ уведомление клиенту авто
              └─> manual_close (менеджер/РОП)

assigned ─> in_progress
              │ (менеджер работает: смотрит каталог, добавляет позиции в КП)
              │
              ├─> awaiting_client_clarification ─> in_progress (клиент ответил)
              │     ▲ запросил менеджер, не KB. Фиксируем в KB suggestion.
              │
              ├─> pending_price_refresh ─> in_progress (по кнопке менеджера)
              │
              ├─> declined_by_us ────────────────────────────────────────► closed_lost
              │     ▲ "не сможем предложить" + reason
              │
              ├─> quoted_partial (отправлено КП на часть позиций)
              │     │ менеджер должен либо дополнить, либо пометить остальные unavailable
              │     │ напоминание T+24ч / T+72ч
              │     ├─> menager_completes ─> quoted
              │     └─> manager_marks_unavailable(items[]) ─> quoted (среди оставшихся = full)
              │
              └─> quoted (полное КП отправлено)

quoted ─> awaiting_client_decision
              │ (детектор клиентских ответов классифицирует входящие)
              │
              ├─> client: "на согласовании" ─> under_review(reminder_date)
              │     После reminder_date — снова attention, повторить пинг
              │
              ├─> client: "перенесём на N мес" ─> postponed_until(date)
              │     В назначенную дату — напоминание менеджеру + предложение
              │     обновить КП (запустить refresh) перед повторным отправлением
              │
              ├─> client: "нужен счёт" ─> awaiting_invoice (менеджер создаёт)
              │     детектор детектирует отправку счёта → invoiced
              │
              ├─> client: "отказ" + причина ───────────────────────────────► closed_lost
              │     reason обязателен (price/timing/competitor/other), сохраняем для аналитики
              │
              ├─> client молчит:
              │     auto reminder T+3, T+7, T+14
              │     после T+30 ─> closed_lost_no_response_to_quote
              │
              └─> manual transition (менеджер видит, что клиент написал по другому каналу
                    — телефон, мессенджер — и руками двигает статус)

invoiced ─> awaiting_payment
              ├─> paid ─> closed_won (опц. ─> shipped, если в корп. базе есть отгрузка)
              ├─> invoice_cancelled ─> in_progress | closed_lost
              ├─> invoice_expired (истёк срок счёта без оплаты):
              │     reminder T-5 дней до истечения, день истечения,
              │     T+7 после истечения ─> closed_lost_invoice_unpaid
              │     либо менеджер инициирует перевыставление ─> in_progress
              │
              └─> manual_close

closed_won  ──► exported
closed_lost ──► exported
              ▲ Все терминальные состояния → ExportRequestJob (см. раздел 8)
```

#### 5.2 Список всех статусов

| Status | Тип | Описание |
|---|---|---|
| `received` | начальный | Только что получили письмо, ничего не сделали |
| `qualifying` | промежуточный | Идёт KB-классификация (L1/L2), оценка тематики |
| `rejected_off_topic` | мягко-терминальный | KB решил «не наша тема», уведомление клиенту ушло. РОП может reopen. |
| `clarifying_with_client` | промежуточный | Не хватает информации, ждём ответ клиента (заявка не назначена менеджеру или назначена в partial-режиме) |
| `assigned` | промежуточный | Назначена менеджеру, ещё не открыта |
| `in_progress` | промежуточный | Менеджер активно работает |
| `awaiting_client_clarification` | промежуточный | Менеджер задал клиенту вопрос (не KB) |
| `pending_price_refresh` | промежуточный | Soft-block: ждём ответы поставщиков |
| `quoted_partial` | промежуточный | Часть позиций отправлена клиенту, остальные нужно дозакрыть |
| `quoted` | промежуточный | Полное КП отправлено клиенту |
| `under_review` | промежуточный | Клиент: «на согласовании», ждём до напоминалки |
| `postponed_until` | промежуточный | Клиент: «перенесём», ждём указанной даты |
| `awaiting_invoice` | промежуточный | Клиент попросил счёт, менеджер должен выставить |
| `invoiced` | промежуточный | Счёт детектирован отправленным |
| `paused` | промежуточный (мета) | **Любая заявка** может быть поставлена на паузу до даты |
| `paid` | промежуточный | Счёт оплачен |
| `closed_won` | терминальный | Успех |
| `closed_lost` | терминальный | Закрыто без продажи (с lost_reason) |
| `exported` | пост-терминальный | Summary выгружен в корп. базу |

**Реанимация закрытых заявок:**
> Если клиент написал после `closed_no_client_response` (или другого «тихого» закрытия) — система **реанимирует ту же заявку**: возвращает её в статус `qualifying` (с пометкой «реанимирована из закрытой»), не создавая дубликат. Это резолюция inbound-парсера — он сматчил входящее письмо к закрытой заявке через References/In-Reply-To или по адресу клиента.
>
> Поле `Request.reanimated_from_request_id` фиксирует историю (если реанимаций несколько). Менеджер видит в карточке маркер «эта заявка ранее закрывалась N дней назад» — для контекста.

**Lost reasons** (taxonomy для аналитики, поле `closed_lost_reason`):
- `off_topic` — нерелевантная заявка
- `no_client_response_to_clarification` — клиент молчал на запрос уточнения
- `no_client_response_to_quote` — клиент молчал после КП
- `client_declined_price` — клиент отказался: дорого
- `client_declined_timing` — клиент отказался: долго
- `client_declined_competitor` — клиент отказался: ушёл к конкурентам
- `client_declined_other` (с текстом)
- `we_cant_offer` — мы сами решили не предлагать (нет возможности поставить, etc.)
- `invoice_unpaid` — счёт не оплачен в срок
- `invoice_cancelled` — счёт отменён
- `manual_other` — закрыто РОПом вручную с комментарием

#### 5.3 Механизм Attention (внимание менеджера)

У каждой заявки есть два метаполя:
- `attention_required_at` (timestamp, nullable) — когда заявка должна снова попасть в фокус.
- `attention_reason` (enum) — почему: `awaiting_client | awaiting_supplier | quote_followup_due | invoice_followup_due | postponed_resume | partial_quote_overdue | sla_breach`

Правила:
- Любой переход статуса (или действие) пересчитывает `attention_required_at` на следующий разумный дедлайн.
- В пуле менеджера — сортировка по `attention_required_at`: просроченные (NOW > attention_required_at) подсвечены красным наверху.
- Если `attention_required_at = NULL` — это либо терминал, либо paused, либо реально ничего не ждём (редко).
- РОП в дашборде видит counters: «X заявок просрочено, Y требуют внимания сегодня».

#### 5.4 Механизм Pause (заморозка)

- Менеджер на любой непомеченной-как-терминальной заявке нажимает «pause until [дата]» с обязательным reason (например: «каникулы у поставщиков до 15.01»).
- Заявка переходит в `paused`, сохраняется `paused_until` и `paused_from_status` (куда возвращать).
- Скрывается из «требуют внимания», не считается в SLA-метриках.
- Cron job `requests:resume-paused` (каждое утро): для всех `paused` где `paused_until <= today`:
  - Возврат в `paused_from_status`.
  - `attention_required_at = now()` → заявка тут же подсвечивается.
  - Запись в audit log: «возобновлена после паузы».
- Pause ограничен по сроку — **по умолчанию 21 день (3 недели)**, по решению клиента. После истечения cap менеджер не может ставить новую паузу без апрува РОПа. Конкретное число — настройка в Settings.

#### 5.5 Non-stuck guarantee (гарантия движения)

> Заявка **не может зависнуть**. Каждое не-терминальное состояние имеет дедлайн. Если дедлайн нарушен — эскалация.

**Дедлайны** (default, настраиваются РОПом):
| Статус | Default deadline | Действие при нарушении |
|---|---|---|
| `received` → `qualifying` | 5 минут (job timeout) | Алерт админу о падении job |
| `qualifying` → `assigned` или `clarifying_with_client` | 15 минут | Алерт админу |
| `clarifying_with_client` без ответа клиента | 3 / 7 / 14 дней | Reminder клиенту, затем auto-close |
| `assigned` → `in_progress` (менеджер не открыл) | 4 рабочих часа | Attention + nudge менеджеру, T+1 день — РОПу |
| `in_progress` без активности | 24 рабочих часа | Attention менеджеру |
| `awaiting_client_clarification` без ответа | 2 / 5 / 10 дней | Reminder клиенту, затем escalation |
| `pending_price_refresh` без ответа поставщика | T+24 / T+72ч | Reminder поставщику |
| `quoted_partial` не дополнено | 24ч / 72ч / 7 дней | Attention менеджеру, T+7 — РОПу |
| `quoted` без реакции клиента | 3 / 7 / 14 / 30 дней | Reminder клиенту, T+30 — auto-close |
| `under_review` (после reminder_date клиента) | На дату клиента + N | Attention менеджеру → пинг клиенту |
| `postponed_until` | На дату клиента | Attention менеджеру → проверить актуальность КП |
| `awaiting_invoice` | 24 рабочих часа | Attention менеджеру |
| `invoiced` без оплаты | T-5 до истечения, день истечения, T+7 | Reminder клиенту, затем close |

**Метрика «застрявших» в дашборде РОПа** — простой запрос: `WHERE attention_required_at < NOW() - interval '24 hours' AND status NOT IN (терминальные, paused)`.

#### 5.6 Завершающее правило

> Менеджер **обязан** довести заявку до одного из терминальных состояний (`closed_won` или `closed_lost`). Все промежуточные пути ведут в терминал — либо через успех (paid → closed_won), либо через отказ с явной причиной.
>
> Запрещено «забыть» заявку. SLA-нарушения видит РОП, в крайнем случае — директорат.

### 6. Уведомления и напоминания (наследуется из LazyLift, расширяется)

Базовый модуль есть в LazyLift (Notification + scheduler). Переиспользуем как есть, **добавляем новые триггеры** под бизнес-сценарии MyLift.

**Полный список триггеров MyLift:**

| Событие | Через сколько | Кому | Канал |
|---|---|---|---|
| Заявка отвергнута как off-topic | сразу | клиент (auto-уведомление) | email |
| Заявка отвергнута как off-topic | сразу | РОП (для аудита) | in-app digest |
| Заявка назначена | сразу | менеджер | in-app + email digest |
| Менеджер не открыл назначенную заявку | T+4 раб. часа | менеджер (нудж) | in-app, push |
| Менеджер не открыл / не двигает | T+1 раб. день | РОП | in-app |
| KB запросил уточнение у клиента | сразу | клиент | email |
| Клиент не ответил на уточнение KB | T+3, T+7 дней | клиент (мягкий пинг), затем auto-close | email |
| Менеджер запросил уточнение у клиента | сразу | клиент | email |
| Клиент не ответил на уточнение менеджера | T+2, T+5, T+10 дней | клиент → менеджер → РОП | email + in-app |
| Поставщик не ответил на refresh | T+24ч, T+72ч | менеджер + закупщик (нудж поставщику) | email копией поставщику + in-app |
| Пришёл ответ от поставщика по refresh | сразу | менеджер | in-app, push |
| Пришёл частичный ответ (часть refused) | сразу | менеджер | in-app |
| Частичное КП не дополнено | T+24ч, T+72ч, T+7д | менеджер → РОП | in-app, email |

| Под `under_review` — наступила дата напоминания | в день | менеджер → клиент | in-app + email клиенту |
| Под `postponed_until` — наступила дата | в день | менеджер: «обновить КП перед повторной отправкой?» | in-app prompt |

| Счёт истёк | в день истечения | менеджер: перевыставить или закрыть? | in-app |
| Счёт истёк + 7 дней без оплаты | T+7 после истечения | auto-close `invoice_unpaid` | — |
| Заявка `paused`: достигнута `paused_until` | в день | менеджер: возврат, требует внимания | in-app + email |
| SLA нарушен (любая просроченная attention) | по таймеру | РОП | in-app |
| Серия из 3+ нарушений SLA одним менеджером | дневной digest | РОП | email digest |

**Реализация:** базовый Laravel `Notification` + scheduler из LazyLift. Дополняем таблицу `request_followups` новыми типами триггеров и обработчиками. Шаблоны писем редактируются в админке РОПом.

### 7. Детектор документов и ответов в исходящих/входящих письмах (новый модуль `app/Services/DocumentDetector/`)

Распознаёт ключевые события в почте, чтобы автоматически двигать жизненный цикл заявки:
- **Outbound от менеджера** — отправка КП, счёта, уточнения клиенту.
- **Inbound от клиента** — ответ на КП с классификацией намерения.

#### 7.1 Outbound: детектор документов

**Триггер:** новое сообщение в `\Sent` любого ящика менеджера, привязанное к `Request` через email pipeline (раздел 1).

**Сигналы для классификации:**
- **Вложения:** имя файла (КП_*.pdf, Счёт_*.pdf, Quote-*, Invoice-*), MIME (application/pdf, xlsx).
- **Текст письма:** ключевые фразы в теле/теме («высылаю КП», «коммерческое предложение», «счёт во вложении», «invoice attached»). AI-классификация (gpt-4o-mini), как в `IncomingMailProcessor`.
- **Структура внутри PDF/XLSX** (опционально, во второй итерации): OCR/extract → проверка на наличие шапки «Коммерческое предложение №...».

**Что детектируем (типы):**
- `quotation_full_sent` → переход `request.status = quoted`, создание `Quotation`
- `quotation_partial_sent` → переход `request.status = quoted_partial`, отметка какие позиции включены (если AI смог разобрать), запуск напоминалок «дополни»
- `invoice_sent` → переход `request.status = invoiced`, создание `Invoice`
- `clarification_to_client` → переход `request.status = awaiting_client_clarification`, **запись KB-suggestion** на основе того, что менеджер счёл недостающим
- `unrelated_communication` — обычная переписка, без триггера статуса

#### 7.2 Inbound: классификатор клиентских ответов

**Триггер:** входящее письмо, привязанное по In-Reply-To/References к существующей `Request` в статусе `quoted` / `quoted_partial` / `under_review` / `postponed_until` / `awaiting_client_clarification`.

AI-классификатор (gpt-4o-mini) определяет намерение клиента:

| Тип намерения | Действие |
|---|---|
| `under_review_acknowledgment` («получили, на согласовании, ответ через N дней») | Перевод в `under_review`. AI извлекает дату или менеджер уточняет. |
| `postponement_request` («перенесём на месяц/квартал») | Перевод в `postponed_until`. AI извлекает дату. |
| `invoice_request` («устраивает, выставите счёт») | Перевод в `awaiting_invoice`, alert менеджеру |
| `decline_with_reason` («дорого / долго / нашли у конкурентов / не нужно») | Предложение перевести в `closed_lost` с подсказанным reason; **финальное решение менеджера** |
| `clarification_response` (если статус был `awaiting_client_clarification`) | Перевод в `in_progress` или возврат в `qualifying` (если KB должен переоценить) |
| `unclear` | Не двигаем статус, alert менеджеру: «пришло письмо, разберись» |

**Важно:** классификатор делает **suggestions**, а не автоматические решения. Менеджер видит UI-prompt: «AI предположил намерение [X], применить?» — одной кнопкой подтверждает или меняет.

#### 7.3 Защита от ложных срабатываний и режим валидации AI

> **Auto-применение AI-решений включается только после прохождения валидационного периода.** На старте все распознавания AI работают в режиме «suggestion + ручное подтверждение». После накопления выборки ≥1000 решений — анализ ошибок: если доля неверных < 0.1% (1 на 1000), AI переводится в auto-режим для соответствующего типа решений.

**Реализация:**
- На каждое распознавание (тип письма, статус КП, intent клиентского ответа) — счётчик `auto_applied`, `manually_overridden`, `error_rate_window`.
- Дашборд РОПа показывает «AI quality score» по каждому типу решений с timeseries.
- Switch «Auto-mode for [type]» в настройках включается **вручную РОПом** при достижении целевого качества.

**Дополнительные защиты (всегда включены):**
- Confidence-порог: ниже определённого уровня — UI-prompt без авто-применения, даже в auto-режиме.
- Audit log: каждое срабатывание детектора пишется в `EmailMessage.detected_artifacts` (jsonb): что распознали, с какой confidence, какое действие предложили/применили.
- Кнопка «откатить» на любом авто-переходе статуса (с записью в `request_state_changes`).
- Метрика «доля откатов» в дашборде РОПа — для tuning порогов и обнаружения деградации модели.

#### 7.4 Сохранение причин отказа для аналитики

При переходе в `closed_lost` фиксируется:
- `closed_lost_reason` (taxonomy enum, см. 5.2)
- `closed_lost_quote` — **цитата из письма клиента** (если отказ детектирован из inbound), точная фраза
- `closed_lost_comment` — свободный текст менеджера (если перевод вручную)
- Ссылка на письмо клиента, в котором обнаружен отказ

> Категории отказа на старте — фиксированные (taxonomy). В будущем планируется AI-анализ накопленных цитат для обнаружения новых паттернов причин («10% отказов содержат фразы про сроки доставки — стоит выделить в отдельную категорию»).

Эти данные → агрегируются на дашборде директората: «структура отказов за период», «средняя сумма потерянных заявок по причинам», «облако цитат» по категории.

### 8. Экспорт в 1С (новый модуль `app/Services/Export/`)

После закрытия заявки (`closed_won` / `closed_lost`) MyLift формирует пакет и пушит в 1С — корпоративную базу клиента. Способ доступа к 1С пока не определён (см. open question), но архитектурно изолирован за интерфейсом `RequestExporterInterface` для будущей замены.

#### 8.1 Маппинг кодов заявок MyLift ↔ 1С

> **Важная особенность:** заявка сначала живёт **только в MyLift** (с собственным `internal_code`, генерируется при создании). Заявка в 1С **создаётся позже** — на финальном этапе (закрытие), либо когда менеджер делает КП/счёт через 1С.
>
> **1С генерирует свой `corp_external_code` при создании**, MyLift узнаёт его post-factum и сохраняет в `Request.corp_external_code`.

Способы синхронизации кода (зависит от возможностей 1С):
- 1С создаёт заявку → возвращает код через API → MyLift сохраняет.
- Менеджер вручную копирует код из 1С в карточку заявки в MyLift (fallback).
- 1С пушит webhook о создании, MyLift матчит по `internal_code` (если 1С сохраняет наш код в своей записи).

#### 8.2 Что экспортируем в 1С

Согласно ответу клиента — **полный пакет** заявки:
- Метаданные: `internal_code`, даты, итоговый статус, причина (если отказ), назначенный менеджер.
- **Вся переписка** (входящая и исходящая) — текст писем, headers, attachments references.
- **Все файлы заказчика** — вложения из писем клиента (фото шильдиков, спецификации, документы).
- Связанные `SupplierOffer` (что прислали поставщики по refresh).
- Метрики обработки.

**Структура** — согласовываем с владельцем 1С (см. open questions).

**Способ передачи** — возможные варианты:
| Способ | Реализация |
|---|---|
| REST API 1С | `POST /requests` или аналог, идемпотентный по `internal_code` |
| Прямая запись в БД 1С | Insert через отдельный connection (с осторожностью!) |
| Файл (XML/JSON) + директория обмена | MyLift пишет файл, 1С подхватывает по cron |
| Webhook от 1С | 1С запрашивает данные у MyLift API |

Архитектурно за интерфейсом `RequestExporterInterface` для замены способа.

#### 8.3 Pipeline

```
ExportRequestJob (триггерится переходом в терминальный статус):
  1. Собрать пакет через RequestPackageBuilder
     (вся переписка, attachments, supplier offers, метаданные)
  2. Идемпотентность: проверить request_exports table по internal_code
     - если уже было успешно — skip
     - если был fail — retry с backoff
  3. Вызвать ExporterInterface::export(package)
  4. Получить corp_external_code (если применимо), сохранить в Request
  5. Записать результат в request_exports (export_id, payload_hash, status, response)
  6. При успехе — request.status = exported, audit log
```

**Ручная инициация** — кнопка «экспортировать сейчас» в карточке заявки (для случаев, когда автоэкспорт упал, и менеджер хочет повторить).

> **Импорт КП и счетов из 1С НЕ делаем** — у клиента нет API для этого. Распознавание этих документов — только через детектор исходящих писем (раздел 7).

### 9. KB-модуль (наследуется из LazyLift с расширением)

- L1/L2 — как в LazyLift, оценка достаточности заявки.

## Открытые вопросы

> Большинство вопросов было закрыто на встрече с представителем MyLift. Ниже — только те, что остаются открытыми и требуют дальнейшей проработки.

### Критические (блокирующие проектирование)

1. **Способ доступа к 1С для чтения каталога** — REST API (если 1С предоставляет), прямой доступ к БД (read replica), CSV/XLSX-импорт по cron, webhook'и от 1С, OData-запросы? Это определяет реализацию `CatalogSyncService`. **Требуется созвон с владельцем 1С/администратором.**

2. **Способ экспорта в 1С** — те же варианты, что для чтения. Желательно зеркально способу чтения, но может быть и иначе. Требует согласования формата структуры с владельцем 1С.

3. **Маппинг кода заявки** — как 1С возвращает свой `corp_external_code` в нашу систему? Через API (если есть response при создании), через webhook, или менеджер копирует руками? От этого зависит надёжность связки.

4. **Куда приходят ответы поставщиков на refresh** — два варианта: специализированный ящик `purchase@..` или адреса конкретных менеджеров. Под вопросом, влияет на:
   - механику парсинга (общий ящик проще для централизованной обработки)
   - на восприятие поставщиком (письмо от менеджера выглядит «теплее»)

### Технические (разрешимы в процессе)

5. **Аутентификация ящиков** — App passwords (просто, но требует от каждого менеджера зайти в настройки Яндекса), или Yandex OAuth (сложнее, но штатнее). На MVP — app passwords, на масштабировании — OAuth.

6. **Формат IMAP-меток в Yandex 360** — какой именно синтаксис custom IMAP keywords корректно отображается в Яндекс веб-клиенте. **Тестируем на боевом ящике в начале Фазы 1**, документируем результат.

7. **Структура summary-payload для экспорта в 1С** — конкретный формат полей, согласовываем с владельцем 1С после ответа на #2.

8. **PII и хранение писем** — храним всё тело письма в БД, или только метаданные + указатель на raw файл? Влияет на размер БД (при 200 заявок/день и сохранении полной переписки + вложений — БД растёт быстро).

9. **Старая переписка** — на старте импортируем последние N дней истории по каждому ящику, или начинаем «с нуля» с момента подключения?

10. **Forward механизм** — пересылаем как новое письмо от ящика MyLift или RFC-style resend? Первое проще, второе чище для получателя.

11. **Полный набор правил маршрутизации** — какие конкретно категории писем команда хочет различать (помимо очевидных: рекламации, бухгалтерия)? Тендеры, маркетинг, partner-запросы — что ещё? **Собираем примеры реальных писем в начале Фазы 1.**

## Жёсткие правила (наследуем из CLAUDE.md LazyLift полностью)

- Не удалять/перезаписывать существующий код без явного запроса.
- Тонкие контроллеры, бизнес-логика в Service classes.
- Form Request для валидации.
- Eloquent > raw SQL, Query Builder для сложных кейсов.
- API Resources, Enums вместо строк.
- Миграции: одна задача = одна миграция, nullable() для опциональных, index() для FK.
- Логирование ошибок с контекстом.
- API ответы: `{ success, data, message }` + явный HTTP-статус.
- Запрещено: `$request->all()`, `dd()` в коммите, hardcoded URL, миграция без `hasColumn`/`hasTable`.

**Дополнительно для MyLift:**
- Все запросы менеджера на просмотр/изменение чужих заявок проходят через политику и логируются.
- Любое изменение `request.assigned_user_id` создаёт запись в `request_assignments` с reason.

## Грабли, которые ждём (с опытом LazyLift + новые от IMAP)

| Грабли | Превентивно |
|---|---|
| Гонка при одновременной приёмке двух заявок на одно оборудование | Lock на equipment_key при назначении (DB advisory lock или транзакция с `FOR UPDATE`) |
| Тестирование sticky-логики на sqlite :memory: | Сразу заводить feature-tests на отдельной testing-postgres БД |
| Перекос: «sticky-менеджер» утонул, остальные простаивают | Workload cap + дашборд перекосов + ручной override РОПа |
| `Notification` в очереди + бизнес-критичные напоминания (счёт истекает) — что если worker лёг | Отдельная очередь `notifications`, мониторинг lag, idempotent re-send по `request_followup.sent_at IS NULL` |
| Менеджер закрыл заявку «лишь бы убрать», SLA метрика портится | Reason при закрытии (declined / not_relevant / won), РОП видит в дашборде разрез |
| **Цена**: одновременно несколько менеджеров запросили refresh одной позиции | Уникальный indexed pending PriceRefreshBatch per (item, supplier); при дубликате — присоединить заявку к существующему batch'у через `request_price_refresh_links` |
| **Цена**: ответ поставщика по части позиций, по части — нет | Помечаем batch как `responses_in` (или вводим отдельный partial), оставшиеся позиции — без offer; менеджер видит, что есть, и решает сам |
| **Поставщик**: один email-адрес обслуживает несколько компаний-поставщиков | На уровне `Supplier` — first-class entity, email — атрибут (может быть shared); матчинг ответа по batch.message_id, не только по From |
| **Reminder**: менеджер шлёт напоминание спустя 1 день, поставщик в этот момент готовит ответ — приходят два письма | Дедуп по Message-ID на уровне ответа, обработка в той последовательности, в которой пришли. Если оба содержат цены — оба становятся SupplierOffer (история). Менеджер видит обоих. |
| **Reminder**: спам-восприятие — после 3-4 напоминаний поставщик начинает игнорировать | Soft-cap на reminder_count (например, 3) с warning в UI «дальше — звонок»; possibly эскалация РОПу |
| **Парсер**: ответ «не работаем с этим брендом, но есть аналог...» — это refused или quoted? | Помечаем как refused, но raw_text в карточке — менеджер видит и решает. Можно добавить тип `partial_with_alternative` если будет частая практика |
| **Парсер**: ответ только текстом «по позиции 3 цена 1500», без явной структуры | Полагаемся на AI с low temperature; в SupplierOffer храним raw_text для аудита — менеджер всегда может проверить распознавание |
| **Workflow**: каталог обновляется ВНЕ MyLift, менеджер ждёт когда там обновят, refresh-ответы есть, но заявка всё равно blocked | Явный UI-намёк: «refresh выполнен, ждём обновления каталога». Возможно — кнопка «уведомить РОПа об обновлении вручную» (вне scope, но коммуникативная подсказка) |
| **CatalogSync**: данные в MyLift отстают от корп. базы — менеджер видит старую цену | Метка «обновлено N мин назад» в UI, ручная кнопка «обновить сейчас» по конкретной позиции, alert если sync не было > X часов |
| **CatalogSync**: корп. база недоступна (даун, сетевые проблемы) | Деградируем gracefully: работаем с последним успешным снапшотом, явно показываем менеджеру «данные на DD.MM», письма всё равно отправляем |
| **Детектор**: ложное срабатывание (письмо «вышлю КП завтра» помечено как `quoted`) | Confidence-порог + UI-кнопка «откатить статус» с записью в audit log; при N откатов — ML-поправка / правила |
| **Детектор**: менеджер прислал клиенту КП через WhatsApp / телефонный звонок, мимо email | MyLift не узнаёт, статус застревает в `in_progress`. Решение: пакетное уточнение через scheduler («статус не изменился N дней — менеджер, что произошло?») |
| **Экспорт**: дублирующая запись в корп. базе при retry экспорта | Идемпотентный ключ по `request.code` + payload_hash в `request_exports`; корп. база upsert по этому ключу |
| **Экспорт**: payload изменился, корп. база уже импортировала старую версию | Логируем разницу payload_hash, посылаем `update`-вариант если корп. база поддерживает; иначе — пометка для ручного разбора |
| **Lifecycle**: KB ошибочно классифицировал тематическую заявку как off-topic | РОП видит весь поток off-topic, может одной кнопкой reopen → quailifying. Фиксируем в `request_state_changes` для tuning классификатора |
| **Lifecycle**: клиент ответил на off-topic уведомление «нет, это ваше», заявка уже закрыта | Inbound-парсер должен матчить ответ к закрытой заявке, alert РОПу: «клиент возражает против отказа, проверь» |
| **Lifecycle**: менеджер злоупотребляет `paused` — паузит проблемные заявки | Cap по дате (max 90 дней?), РОП видит «долго запауженные» в дашборде, метрика «доля paused от общего» |
| **Lifecycle**: классификатор клиентского ответа сработал ошибочно (`under_review` вместо отказа) | UI всегда показывает «AI suggestion» с кнопкой подтвердить/изменить; auto-apply только при confidence > порога |
| **Attention**: менеджер не видит просроченные заявки (заваленый pool) | Сортировка по `attention_required_at` ASC + цветовая индикация + бейджик с числом просроченных в навигации |
| **Counter**: одна и та же позиция в `quoted_partial` и в `pending_price_refresh` одновременно — двусмысленно | Запрещаем одновременность: если запрашиваем refresh, отменяем `quoted_partial` для этих позиций (или разбиваем на под-заявки — open question) |
| **Auto-close**: клиент молчит 30 дней после КП → авто-закрытие, но клиент мог быть в отпуске | Final reminder за 3 дня до auto-close, явное «если не ответите до DD.MM, мы закроем как нет ответа»; клиент может ответить и заявка реактивируется |
| **IMAP**: UIDVALIDITY поменялась — всё, что мы знали по UID, протухло | Хранить пару `(uidvalidity, last_uid)` per folder; при смене — full resync с дедупом по `Message-ID` |
| **IMAP**: соединение рвётся, polling job падает | Retry с backoff, не валим whole sync, alert если N подряд неудач |
| **IMAP**: одно и то же письмо в Inbox получателя и в Sent отправителя — два EmailMessage, но один Request | Дедуп по `Message-ID` на уровне Request-связки, но `EmailMessage` per (mailbox, folder) — оба сохраняются |
| **Yandex**: специфика папок «Входящие/Отправленные» по-русски, кодировки | Использовать IMAP special-use flags `\Inbox`/`\Sent`, не имена. Тестить на тестовом ящике с реальной перепиской |
| **Yandex**: лимиты на одновременные IMAP-сессии (точное число надо проверить) | Один долгоживущий клиент per mailbox, не открывать на каждый poll |
| **App passwords**: менеджер сменил пароль/удалил app password — IMAP падает молча | Health-check job ежедневно, статус ящика в админке РОПа, alert при проблеме |
| Менеджер ответил клиенту из Яндекс веб-клиента → мы это узнаём только при следующем polling Sent | Polling Sent делать с тем же приоритетом, что Inbox; для real-time — IDLE на Sent тоже |
| Письмо с большим вложением (фото шильдиков лифта) — память worker'а | Streaming download attachment, лимит размера, ассинхронная Vision-обработка отдельным job |
| **Routing**: правило настроено криво, заявки ушли в бухгалтерию | Превью на реальном письме перед сохранением правила; метрика «forward'ов в нетипичные адреса»; легко откатить (правило toggle, лог в RoutedMail) |
| **Routing**: одно письмо матчится под несколько правил | Поле `priority` + `is_terminal`; иначе порядок не определён, лог в RoutedMail отражает только первое сработавшее |
| **Routing**: правило `forward → external` уходит в спам у получателя (не привычный sender) | Проверять SPF/DKIM при настройке forward; либо forward через ящик MyLift, либо через native IMAP message redirect (resend) — это open question |
| **Labels**: Yandex IMAP не поддерживает кастомный формат флагов как ожидалось | Test на боевом ящике перед написанием. Fallback — использовать существующие user-flags Yandex (`$Forwarded`, custom keywords). Документируем в `MAILBOXES.md` |
| **Labels**: после смены статуса Request не обновился label на письме | Job `mail:sync-labels` по событию RequestStateChange; идемпотентный (не падает если письмо уже с этим label) |
| **Labels**: пользователь вручную убрал нашу label в Яндекс клиенте | Не пытаемся восстанавливать (это намеренное действие); следующий FETCH покажет отсутствие — фиксируем в EmailMessage.label_state |
| **Forward**: пересланное письмо вернулось как bounce | Парсим bounce → alert РОПу + flag в RoutedMail; правило временно деактивируется до выяснения |
| **Privacy**: forward пересылает на external email вложения с PII | Конфигурация: «не форвардить вложения для типа X», или «форвардить только текст письма» |

## Фазы реализации

> **Решение клиента:** в MVP — только чтение почты и классификация. Всё остальное — последующие фазы. Это снижает риск и даёт быстрый ROI.

**MVP (Фаза 1) — чтение и классификация писем (4-6 нед)**
- Скелет проекта на стеке (Laravel + PostgreSQL + Livewire), миграции базовых сущностей.
- Auth + 4 роли (менеджер / РОП / секретарь / директорат) + Spatie Permission.
- **Email integration**: подключение всех общих и личных ящиков (через настраиваемый список в админке, не хардкод), polling Inbox + Sent.
- **MailDispatcher + MailRoutingRule** — настраиваемые правила, UI на `/dashboard/mail-rules`, превью на реальных письмах.
- **MailLabelService** — постановка IMAP custom labels (с именем менеджера для заявок).
- **MailForwarder** — пересылка по правилам.
- AI-классификация типа письма (gpt-4o-mini): request / reclamation / accounting / general / spam / other.
- Базовое создание `Request` для заявок (без сложной обработки — просто запись + назначение по упрощённому правилу).
- Наблюдение за качеством AI: дашборд РОПа с метриками распознавания.
- Статус интеграций, health-check ящиков.

> К концу Фазы 1: команда видит, что система разбирает поток почты, помечает в Яндексе, форвардит куда надо. Заявки попадают в систему — но обрабатываются ещё минимально.

**Фаза 2 — распределение, KB и пул менеджера (3-4 нед)**
- Полноценная KB-обработка заявок (L1/L2 — оценка качества и тематики).
- Auto-rejection нерелевантных + reopen РОПом.
- `AssignmentService` — sticky по `catalog_item_id` + load balancing.
- Личный пул менеджера (Livewire).
- История назначений + UI переподчинения.
- Менеджер «недоступен» с переподчинением заявок.
- Дашборд РОПа v1 (заявки, нагрузка по менеджерам).
- Базовые напоминания (заявка не открыта, нет активности).

**Фаза 3 — каталог и refresh (3-4 нед)**
- Интеграция с 1С на чтение каталога (зависит от ответа на open question про способ).
- Матчер позиций на каталог.
- `Supplier` + интерфейс с текстовым описанием ассортимента + AI-сервис построения матрицы.
- `PriceRefreshService` + dispatch к релевантным поставщикам + парсер ответов + сохранение `SupplierOffer` (quoted/refused).
- UI «Refresh цен» в карточке заявки + общий пул `/dashboard/supplier-pool` с напоминаниями.

**Фаза 4 — жизненный цикл, детектор и follow-up (4-5 нед)**
- Полная state machine (все статусы из 5.2).
- `RequestStateChange` audit log.
- Attention-механизм + Pause (cap 21 день).
- DocumentDetector (КП full/partial, счёт, уточнение в outbound).
- Inbound-классификатор клиентских ответов (under_review / postponed / decline / invoice_request).
- Auto-close по таймаутам клиентского молчания + last reminder.
- Реактивация закрытых заявок при ответе клиента.
- Расширенные напоминания (вся таблица из 6.2).
- Шаблоны писем (один универсальный для всех типов с переменными).

**Фаза 5 — экспорт в 1С (2-3 нед, зависит от способа)**
- `ExportService` — реализация под выбранный способ.
- Маппинг `internal_code` ↔ `corp_external_code`.
- Полный пакет экспорта (вся переписка + файлы клиента).
- Идемпотентность + retry + ручная инициация.

**Фаза 6 — продвинутая аналитика и валидация AI (1-2 нед)**
- Дашборд директората.
- KB-инструменты для директората (расширенная разметка).
- AI quality dashboard: метрики качества каждого классификатора + переключатели «auto-mode».
- Аналитика причин отказа (категории + цитаты, без AI-разбора пока).
- Опционально — переход с polling на IMAP IDLE если задержка стала критичной.

> Расчёт ЗП менеджеров и распределение заслуг при переназначении — **вне scope проекта**, не реализуется ни в одной фазе.

## Точки входа (после старта проекта)
